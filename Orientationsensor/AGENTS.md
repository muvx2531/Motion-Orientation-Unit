# AGENTS.md

## Project

This repository is for the **Motion Orientation Unit**, an STM32-based IMU/AHRS firmware project.

The target device reads motion and environmental sensors, performs calibration, estimates orientation, and sends real-time data to a computer through USB FS.

Current hardware context:

- STM32 microcontroller
- MPU6500 accelerometer + gyroscope
- LIS3MDL magnetometer
- DPS310 barometric pressure sensor
- L76L-M33 GNSS module
- USB FS connection to PC

Current project focus:

- Sensor drivers
- Sensor calibration
- Orientation output
- Relative altitude
- USB communication
- Firmware architecture
- Test and validation

The current phase is **orientation and relative altitude**, not full GNSS-based navigation or accurate local XYZ position tracking.

## Working Rules for Codex

Before making changes, understand the current project stage and existing files.

Do not silently invent missing hardware details such as pins, SPI/I2C selection, interrupt lines, DMA channels, timers, USB class, flash layout, or packet format.

If a requirement is unclear, ask a focused question or make a clearly marked assumption.

Do not put all logic in `main.c`.

Do not remove existing user code sections generated by STM32CubeMX unless explicitly asked.

Do not rewrite the architecture without explaining the impact.

Do not start sensor fusion work before sensor driver and calibration requirements are clear.

Do not mix human-readable debug text with binary USB streaming unless the protocol explicitly supports it.

## Firmware Architecture

Prefer a layered firmware structure:

- `Core/` or board support layer
- sensor driver layer
- sensor service layer
- calibration layer
- sensor fusion / AHRS layer
- USB protocol layer
- application layer

Recommended module responsibilities:

- **Driver layer**: register access, initialization, raw data read, device ID check, low-level error status
- **Service layer**: sampling schedule, timestamping, unit conversion, sensor health management
- **Calibration layer**: calibration procedure, validation, storage, application of correction values
- **Fusion layer**: quaternion-based orientation estimation and optional Euler conversion
- **Protocol layer**: USB packet framing, commands, responses, status, CRC/checksum
- **Application layer**: system state machine and high-level behavior

## Coding Style

Prefer clear C code for embedded firmware.

Use STM32 HAL by default unless the task explicitly asks for LL, bare-metal, or mixed HAL/LL code.

Keep hardware-specific code separated from algorithm code.

Use explicit error codes instead of ambiguous boolean returns when practical.

Use fixed-width integer types such as `uint8_t`, `uint16_t`, `uint32_t`, and `int16_t` for register-level code.

For public APIs, prefer predictable naming:

- `module_init()`
- `module_deinit()`
- `module_read_raw()`
- `module_read_scaled()`
- `module_configure()`
- `module_self_test()`
- `module_get_status()`

Add comments only where they explain important engineering decisions, hardware constraints, calibration assumptions, or non-obvious timing behavior.

## Sensor Driver Rules

Each sensor driver should clearly define:

- bus interface
- register definitions
- initialization sequence
- device ID check
- configuration options
- raw data structure
- scaled data structure
- error/status reporting
- timeout behavior
- optional self-test

For MPU6500, always clarify or preserve:

- SPI or I2C mode
- gyro full-scale range
- accelerometer full-scale range
- output data rate
- DLPF setting
- FIFO usage or direct register reads
- interrupt usage
- timestamp requirement
- sensor mounting orientation

## Calibration Rules

Calibration is a core requirement and must not be skipped.

The firmware should be designed to support:

- gyroscope zero-rate offset calibration
- accelerometer offset calibration
- accelerometer scale calibration
- magnetometer hard-iron calibration
- magnetometer soft-iron calibration
- barometer reference pressure or reference altitude calibration
- calibration data storage
- calibration data validation
- calibration status reporting

Calibration data storage should include:

- magic value
- version
- sensor identifier
- calibration values
- validity flag
- CRC/checksum

Do not assume calibration must run every boot. Clarify whether each calibration is factory, user-commanded, or startup calibration.

## Orientation and Sensor Fusion Rules

Use quaternion as the preferred internal representation for orientation.

Euler angles may be provided for display, logging, or debugging.

Do not design full navigation, global coordinates, or accurate local XYZ position tracking unless explicitly requested.

If GNSS is unavailable or not ready, keep the core system focused on orientation and relative altitude.

If the IMU is not located at the mechanical center of rotation, clarify whether lever-arm compensation is required. For orientation-only output, it is usually not required. For accurate linear acceleration at a different reference point, it may be required.

## USB Communication Rules

USB FS is used for communication with a PC.

USB CDC is acceptable for early development unless another USB class is requested.

Before implementing the protocol, clarify:

- USB class: CDC, HID, vendor class, or composite
- stream direction: device-to-PC only or bidirectional
- output format: CSV, JSON, binary packet, or custom framed packet
- update rate
- timestamp format
- command format
- calibration command support
- checksum or CRC requirement

Recommended command categories:

- device information query
- sensor status query
- start/stop streaming
- set output rate
- start calibration
- read calibration
- save calibration
- erase calibration
- reset device

## System State Machine

Prefer a clear application state machine:

- `BOOT`
- `SENSOR_INIT`
- `CALIBRATION_LOAD`
- `IDLE`
- `STREAMING`
- `CALIBRATING`
- `FAULT`

Fault behavior should be explicit for:

- sensor not detected
- bad WHO_AM_I value
- bus timeout
- invalid calibration data
- USB disconnect
- packet overflow
- watchdog reset

## Testing Requirements

After code changes, provide practical test steps.

Relevant tests include:

- build test
- MPU6500 WHO_AM_I test
- LIS3MDL ID test
- DPS310 ID or coefficient read test
- USB enumeration test
- raw sensor streaming test
- stationary gyro bias test
- accelerometer gravity magnitude test
- magnetometer rotation test
- calibration save/load test
- invalid calibration CRC test
- packet checksum test
- long-running stability test
